Nacos 报错 parse data from Nacos error

2020-10-02  626 字  阅读时间 3 分钟

使用 Nacos 中踩的一个小坑

问题背景

由于现在的大型应用架构都是分布式、微服务云云。平常开发时弱鸡笔记本难以负担大量的服务和集群，于是乎在家里整了个机架服务器，用阿里云的学生机做内网穿透。平常服务就都搭在这个机架服务器上

今日在服务器上使用 Docker 搭建完 Nacos 后，将应用程序的 Nacos 服务端切换过去时，程序启动失败了，查看日志大致如下：

2020-10-02 00:40:42.717 ERROR 7171 --- [           main] c.a.c.n.c.NacosPropertySourceBuilder     : parse data from Nacos error,dataId:nacos-test,data:������+-N-�KLO�R0��*q�s�����.��\|����,

org.yaml.snakeyaml.reader.ReaderException: special characters are not allowed

大概就是从 Nacos 中读取到的配置是一串乱码，配置文件解析失败了

解决过程

遇到乱码那么第一反应当然是编码问题，尝试切换了下编码后无果，于是在笔记本上重新搭建了一个 Nacos，测试后是可以正常使用的。那么就可能是通过阿里云机子内网穿透时这一步出现了问题，我将两边Nacos 的 HTTP 请求抓了下包，在 Response Header 处发现了端倪

// 经过阿里云机子内网穿透后的
Connection: keep-alive
Content-Encoding: gzip
Content-Language: zh-CN
Content-Type: text/html;charset=UTF-8
Date: Thu, 01 Oct 2020 16:45:00 GMT
Last-Modified: Tue, 04 Aug 2020 11:26:49 GMT
Server: nginx
Transfer-Encoding: chunked
Vary: Accept-Encoding

// 直接搭在笔记本上的
Accept-Ranges: bytes
Content-Language: zh-CN
Content-Length: 2624
Content-Type: text/html;charset=UTF-8
Date: Thu, 01 Oct 2020 16:28:22 GMT
Last-Modified: Tue, 04 Aug 2020 11:26:49 GMT

可以发现 Accept-Ranges: bytes 和 Content-Length 在出问题的请求中是没有的，Accept-Ranges: bytes 是表示服务端支持分段下载，这样客户端和浏览器（支持的话）就可以断点续传或多线程下载

而出问题的的响应头有 Content-Encoding: gzip 表示使用了 gzip 压缩，那么问题就很显然了，Nginx 错误配置导致 Nacos 服务端的响应使用了 gzip 压缩，而 Nacos 客户端无法处理 gzip ，于是出现了一坨乱码

把 Nginx 对应代理的 gzip 关闭后，问题解决（或者可以在代理中设置 Accept-Encoding 标识不支持 gzip ）

Accept-Ranges: bytes
Connection: keep-alive
Content-Language: zh-CN
Content-Length: 2624
Content-Type: text/html;charset=UTF-8
Date: Thu, 01 Oct 2020 16:57:45 GMT
Last-Modified: Tue, 04 Aug 2020 11:26:49 GMT
Server: nginx
